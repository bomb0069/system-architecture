Module 01

System Analyst
คืออะไร?

SA คือ "สะพาน" ระหว่างโลก Business กับโลก Technology — แปลความต้องการของธุรกิจให้กลายเป็น Requirement ที่ Developer สร้างระบบได้จริง

// Core Definition
💡
Systems Analyst (SA) คนที่ใช้เทคนิคการวิเคราะห์และออกแบบเพื่อแก้ปัญหาธุรกิจด้วย IT — ทำหน้าที่เป็น Change Agent ที่ระบุว่าองค์กรต้องปรับปรุงอะไร ออกแบบระบบให้รองรับการเปลี่ยนแปลงนั้น และฝึกให้คนอื่นใช้ระบบได้
// หน้าที่หลัก 6 ด้าน
1. ทำความเข้าใจธุรกิจ (Business Understanding) +
ศึกษา Domain ของธุรกิจให้เข้าใจว่า Business ทำงานอย่างไร เข้าใจ Pain Point และเป้าหมายของแต่ละ Stakeholder วิเคราะห์ระบบเดิมที่มีอยู่ (As-Is) ว่ามีข้อจำกัดอะไรบ้าง
2. รวบรวมและวิเคราะห์ Requirement +
จัด Workshop / Interview กับ Business Owner, End User, IT Team แยกแยะ Business Requirement vs Functional Requirement vs Non-Functional Requirement ตรวจสอบว่า Requirement ขัดแย้งกันหรือไม่ และ Prioritize
3. ออกแบบ Solution +
ออกแบบ Process Flow ใหม่ (To-Be) ออกแบบ Data Flow, Entity Relationship, System Architecture ในระดับ High-level เลือกและเสนอแนวทาง Solution ที่เหมาะสม
4. เขียน Specification เอกสาร +
BRD (Business Requirements Document) — สิ่งที่ Business ต้องการ
FRS/SRS (Functional/System Requirements Specification) — สิ่งที่ระบบต้องทำ
Use Case / User Story — อธิบาย Behavior จากมุมมองผู้ใช้
Wireframe / Mockup — ภาพคร่าวๆ ของหน้าจอ
5. สนับสนุนทีม Development +
ทำหน้าที่เป็น Single Point of Contact ระหว่าง Business กับ Dev/QA ตอบคำถาม Dev เมื่อ Requirement ไม่ชัดเจน Review Test Case กับ QA ว่าครอบคลุม Requirement หรือไม่
6. UAT Support & Go-Live +
ช่วย Business ทำ UAT (User Acceptance Testing) บริหาร Defect แยกแยะว่าเป็น Bug หรือ Change Request ยืนยันว่าระบบตรงตาม Requirement ก่อน Go-Live และส่งมอบ Document ครบชุด
// กระบวนการทำงาน (Process Flow)
1. Initiation └─ รับ Project Brief → ระบุ Stakeholder → วางแผน Requirement Gathering 2. Requirement Gathering └─ Interview / Workshop / Observation → รวบรวม As-Is Process 3. Requirement Analysis └─ วิเคราะห์ Gap (As-Is vs To-Be) → Prioritize → ตรวจหา Conflict 4. Solution Design └─ ออกแบบ To-Be Process → Data Model (Logical) → Wireframe 5. Documentation └─ เขียน BRD / FRS / Use Case → Review → Baseline Requirement 6. Development Support └─ ตอบคำถาม Dev → บริหาร Change Request → ติดตาม Progress 7. Testing Support └─ Review Test Case → ช่วยทำ UAT → บริหาร Defect List 8. Go-Live & Closure └─ Sign-off → Training Material → ส่งมอบ Document ครบ
// Skills ที่ SA ต้องมี

Soft Skills

Communication, Facilitation, Critical Thinking, การเจรจาต่อรอง, การฟังอย่างตั้งใจ

Analytical Skills

วิเคราะห์ Process, Data Modeling, Root Cause Analysis, Systems Thinking

Technical Knowledge

เข้าใจ Database, API, System Architecture เบื้องต้น, BPMN, UML

🎯
กฎทองของ SA SA ที่ดีต้อง "ถามคำถามที่ถูกต้อง" ไม่ใช่แค่จดสิ่งที่ Business พูด เพราะ Business มักบอก "วิธีแก้" แต่ SA ต้องหา "ปัญหาที่แท้จริง"

Module 03

SA vs BA
vs Architect

ขอบเขตระหว่างสามบทบาทนี้ทับซ้อนกันมาก — แต่มีหลักการแยกได้ชัดเจนถ้าเข้าใจว่าแต่ละคนโฟกัสที่ What, Why หรือ How

// เปรียบเทียบ 3 บทบาท
ด้าน System Analyst Business Analyst Solution Architect
โฟกัส What & Why (System) What & Why (Business) What & How (High-level)
มุมมอง Business + System Business Process Business + Technology
Output หลัก BRD, FRS, Use Case, Data Model BRD, Process Map, User Story Architecture Diagram, ADR
Architecture High-level เท่านั้น ไม่เกี่ยวข้อง เป็นงานหลัก
เทียบได้กับ Bridge SA↔BA↔Dev Voice of Business Technical Decision Maker
// ประวัติ: Software Architect เกิดก่อนหรือหลัง SA/BA?
🏛️
คำตอบ: เกิดหลัง SA แต่พร้อมๆ กับ BA (วัดที่ Job Title)
ถ้าวัดที่ Concept เกิดปี 1968 (หลัง SA 1956) — ถ้าวัดที่ Job Title เกิดปลาย 1990s–2001 ซึ่งพร้อมกับ BA ที่กำลังแยกตัวออก
1956 SA เกิด — ทำทุกอย่างรวมถึง Architecture ในคนเดียว 1968 Dijkstra: "Software Architecture" เป็น Academic Concept ครั้งแรก 1972 Parnas เพิ่มแนวคิด Information Hiding / Modularity 1970s–80s SA ยุค Peak — SA ยังทำ Architecture งานด้วยตัวเอง 1990s ระบบซับซ้อนขึ้น → SA เริ่มแตกออก → Architect เริ่มแยกตัว 1994 Gang of Four Design Patterns — ภาษากลางของ Architect เกิด 1996 Shaw & Garlan: Software Architecture เป็น Formal Discipline 1999–2001 "Software Architect" กลายเป็น Job Title อย่างเป็นทางการ 2000 IEEE 1471-2000 — มาตรฐาน Software Architecture ฉบับแรกของโลก 2003 IIBA ก่อตั้ง → BA เป็น Formal Profession แยกจาก SA ชัดเจน
💡
Insight สำคัญ ก่อนที่ Software Architect จะเป็น Role แยก — SA คือคนทำ Architecture อยู่แล้ว ในยุค 1970s–80s นี่คือเหตุผลว่าทำไม SA ไทยปัจจุบันยังทำ Technical Design ด้วย มันไม่ใช่ "ผิดขนบ" แต่คือ Legacy ของยุคที่ Role ยังไม่แยก และทีมเล็กไม่มี Headcount พอสำหรับ Dedicated Architect
// Architecture — SA ทำแค่ไหน?
✅
SA ต้องทำ Logical Architecture, Context Diagram / System Boundary, Data Architecture (Logical), Integration Architecture — กำหนด Pattern การเชื่อมต่อ เช่น REST API
🚫
SA ไม่ต้องทำ เลือก Technology Stack, ออกแบบ Infrastructure, กำหนด Code Pattern, ออกแบบ Database Schema ระดับ Physical
// 3 แบบที่พบในองค์กรจริง
แบบที่ 1 — มี Solution Architect แยกต่างหาก (Enterprise ใหญ่) +
SA → เขียน Requirement + High-level Context เท่านั้น
Solution Architect → ออกแบบ Architecture เต็มรูปแบบ
Software Architect → ลงรายละเอียด Code Pattern และ Technical Design

พบใน Enterprise ขนาดใหญ่ที่มี Resource เพียงพอสำหรับ Specialist แยกย่อย
แบบที่ 2 — SA ทำ Architecture ด้วย (พบมากที่สุด) +
SA → Requirement + Logical Architecture + Integration Design
Dev/Tech Lead → Physical Design + Implementation

พบใน SME และทีมขนาดกลาง — เป็นแบบที่พบมากที่สุดในตลาดไทย
แบบที่ 3 — SA คือ Product Owner ใน Agile +
SA/PO → User Story + Acceptance Criteria + Backlog Management
Tech Lead → Architecture Decision

พบในทีม Startup หรือ Agile เต็มรูปแบบที่ไม่มี Formal Role ชัดเจน
📐
กฎง่ายๆ ในการแบ่งขอบเขต (ตามตำราสากล) ถ้า Architecture Decision นั้นส่งผลต่อ Requirement หรือ Scope → SA ต้องรู้และมีส่วนร่วม
ถ้าเป็นแค่ Implementation Detail → ปล่อยให้ Architect และ Dev ตัดสินใจ

⚠️ แต่ในบริบทไทยอาจต่างออกไป — ดู Module ถัดไป

ทีม Dev มักไม่ใหญ่พอ

บริษัท SME และ Outsource ในไทยส่วนใหญ่ไม่มี Headcount แยก Solution Architect, SA, Tech Lead ไว้คนละคน — คนที่ฉลาดที่สุดต้องทำทุกอย่าง

Career Path ของ Dev ในไทย

เส้นทางที่ชัดเจนที่สุดในไทยคือ Developer → Senior Dev → SA/Tech Lead → PM ทำให้ SA มักมี Coding Background มาก่อน

ค่าแรงและ Efficiency

วัฒนธรรมองค์กรไทย

การที่ SA "อยู่ใต้" Dev Lead แต่ทำหน้าที่นำ Dev ทีมในทางปฏิบัติ สะท้อน Hierarchy ที่ยังนิยมมี Title Seniority ชัดเจน

Module 05

ทำไม SA
ไม่มี BoK?

BA มี BABOK, PM มี PMBOK, SE มี SWEBOK — แต่ SA ไม่มี "SA BoK" ที่เทียบเท่า เหตุผลนั้นน่าสนใจมาก

// ทำไมถึงไม่มี?
BA (IIBA/BABOK) เจ้าของชัดเจน — IIBA ↘ SA ◄── อยู่ตรงกลาง ไม่มีเจ้าภาพชัดเจน ↗ SE (INCOSE/SEBoK) เจ้าของชัดเจน — INCOSE + IEEE

ชื่อตำแหน่งไม่ Standard

บางองค์กรเรียก SA, บางที่เรียก BA, Business Systems Analyst, Functional Analyst — คนละชื่อแต่งานเหมือนกัน ยากจะกำหนดขอบเขตชัดเจน

Scope ทับซ้อนมาก

งาน SA ทับกับ BA, Solution Architect, Product Owner จนแยกออกจากกันยาก IIBA ดึง BA ออกไป INCOSE ดึง SE ออกไป ที่เหลือตรงกลางไม่มีใครอ้างสิทธิ์

ไม่มีองค์กรกลางที่แข็งแกร่งพอ

ไม่มี Professional Body ที่ Focus เฉพาะ SA มีอำนาจและทรัพยากรพอที่จะสร้างและ Maintain Standard ระดับนั้น

Market ไม่ใหญ่พอ

Certification ขายได้ยากกว่า CBAP หรือ PMP เพราะ Identity ของ SA ไม่ชัด ทำให้ ROI ในการสร้าง Standard ต่ำเกินไป

// สิ่งที่ใกล้เคียงที่สุดที่มีอยู่
SEBoK — Systems Engineering Body of Knowledge +
เจ้าของ: INCOSE (International Council on Systems Engineering) + IEEE

เน้น Systems Engineering ซึ่งใหญ่กว่า SA มาก ครอบคลุม Hardware, Software, Process ทั้งหมด เหมาะกับ SA ที่ทำงานระดับ Enterprise หรือ Critical Systems

ข้อดี: เป็น Open Access อ่านได้ฟรีที่ sebokwiki.org และ IEEE เป็น Co-owner ทำให้มี Academic Peer Review มากกว่า BABOK
SWEBOK — Software Engineering Body of Knowledge +
เจ้าของ: IEEE Computer Society

ครอบคลุม Requirements Engineering ซึ่งเป็นส่วนหนึ่งของงาน SA โดยตรง เน้น Technical มากกว่า Business เป็นมาตรฐาน ISO ด้วย (ISO/IEC TR 19759)

Module 06

SA มาก่อน BA
และ Architect

SA เกิดในยุค 1956 ครองตลาดกว่า 30 ปี ทั้ง BA และ Software Architect เป็นบทบาทที่แตกออกมาจาก SA เมื่อระบบซับซ้อนจนคนเดียวทำไม่ไหว

// Timeline ที่ตรวจสอบแล้ว
1956
SA — Birth · New York Times, 1963
"Systems Analysis" ปรากฏครั้งแรก
Herbert D. Benington นำเสนอ Phased Software Development ที่ Symposium on Advanced Programming Methods for Digital Computers — นับเป็นต้นกำเนิดการคิดเชิงระบบใน Computing
1967
SA — Growth · Harvard Business Review 1968
SA เติบโตอย่างก้าวกระโดด
จำนวน Programmer + SA เพิ่มจาก 317,000 คน (1967) เป็น 560,000 คน (1970) สะท้อนการระเบิดตัวของ Computing ในองค์กรธุรกิจ
1970
Milestone · Royce 1970 / IEEE WESCON
Royce Paper — เข้าใจผิดกันมา 50 ปี
Royce ตีพิมพ์บทความที่ถูกเรียกว่าต้นกำเนิด Waterfall — แต่ Royce เองไม่ได้สนับสนุน Sequential Waterfall และไม่ได้ใช้คำนี้เลย คำว่า "Waterfall" ถูกใช้ครั้งแรกโดย Bell & Thayer ในปี 1976
1968–72
Software Architecture — Concept · Dijkstra / Parnas
"Software Architecture" เป็น Academic Concept ครั้งแรก
Edsger Dijkstra (1968) วางรากฐานแนวคิด Structured Software และ David Parnas (1972) เพิ่มแนวคิด Information Hiding และ Modularity — ยังเป็นแค่ Research Concept ยังไม่มี Job Title "Architect" ในยุคนี้ SA คือคนที่ทำงาน Architecture อยู่แล้ว
1970s–80s
SA — Peak · CMS Wire / modernanalyst.com
ยุครุ่งเรืองสูงสุดของ SA
DFD, ERD กลายเป็นเครื่องมือมาตรฐาน SA ทำหน้าที่ครบในคนเดียว — part planner, part consultant, part subject matter expert, part technology translator ยังไม่มีคำว่า "Business Analyst" ในสาย Software
1990s
SA → BA · CMS Wire / modernanalyst.com
SA แตกออกเป็นหลายบทบาท
ระบบซับซ้อนขึ้น ERP เข้ามา SA แตกเป็น Business Analyst, Computer Systems Analyst, Project Manager, Risk Analyst — BA เป็นหนึ่งในผลที่แตกออกมาจาก SA โดยตรง และ Software Architect เริ่มแยกออกเป็น Role เฉพาะในช่วงนี้เช่นกัน
1994
Software Architecture — Patterns · Gang of Four
Design Patterns (1994) — ภาษากลางของ Architect เกิดขึ้น
หนังสือ "Design Patterns: Elements of Reusable Object-Oriented Software" โดย Gang of Four เปิดทางให้ Architect มีภาษากลางในการสื่อสาร Architecture Decisions ตามมาด้วย Martin Fowler "Patterns of Enterprise Application Architecture" (2002)
1996
Software Architecture — Discipline · Shaw & Garlan, CMU
Shaw & Garlan: Software Architecture เป็น Formal Discipline
Mary Shaw และ David Garlan แห่ง Carnegie Mellon เขียนหนังสือที่กำหนดนิยาม Software Architecture อย่างเป็น Formal Discipline ครั้งแรก — แนะนำแนวคิด Components, Connectors, และ Architectural Styles
1999–2001
Software Architecture — Job Title · IEEE 1471-2000
"Software Architect" กลายเป็น Job Title + IEEE 1471 Standard
OOP แพร่หลาย → Application ใหญ่ขึ้น → ต้องการคน High-level Design โดยเฉพาะ "Software Architect" เริ่มปรากฏใน Job Posting อย่างเป็นทางการ พร้อมกับ IEEE 1471-2000 มาตรฐาน Software Architecture ฉบับแรกของโลก (ต่อมาเป็น ISO/IEC 42010)
2001
Milestone · Agile Alliance
Agile Manifesto ลงนาม (ที่ Snowbird, Utah)
Scrum/XP เริ่มพัฒนาช่วง 1994–1995 แต่ Agile Manifesto ลงนามอย่างเป็นทางการในกุมภาพันธ์ 2001 ทำให้บทบาทที่คุยกับ Business ได้รับความสนใจมากขึ้น
2003
BA — Official · IIBA.org
IIBA ก่อตั้ง — BA เป็นอาชีพอย่างเป็นทางการ
International Institute of Business Analysis ก่อตั้งที่ Toronto, Canada — เริ่มพัฒนา BABOK และ CBAP Certification ทำให้ BA มี Identity แยกจาก SA อย่างเป็นทางการ
2006–2015
BA — Mature · IIBA.org
BABOK v1.6 (2006) → v2 (2009) → v3 (2015)
BA มี Standard ครบชุด BABOK v3 ขยายครอบคลุม Agile, Business Intelligence, Business Architecture — ยังคงเป็น Version ล่าสุดจนถึงปัจจุบัน
Now
Current · BLS.gov 2024
SA, BA และ Architect ยังทับซ้อนกันมากในทางปฏิบัติ
BLS ยังติดตาม Computer Systems Analyst แยกจาก BA (Median Wage $103,790/ปี, เติบโต 9% ในช่วง 2024–2034) — ในทีมขนาดกลาง SA มักทำ Architecture ด้วย และ Software Architect บางคนก็ทำ SA งานควบคู่ DraftKings Engineering: "Software Architecture is a function, not a rank"

SA ควรทำ Architecture ในระดับใด (ตาม Global Standard)?
Physical Architecture — ลงถึง Database Schema และ Server Config
Logical Architecture — Context Diagram, System Boundary, Integration Pattern ระดับ High-level
SA ไม่ควรยุ่งเกี่ยวกับ Architecture เลย
ทุกระดับเท่ากับ Solution Architect
✅ ถูกต้อง! SA ทำ Logical Architecture เท่านั้น — กำหนดว่าระบบมี Component อะไร ทำงานร่วมกันอย่างไร Data ไหลจากไหนไปไหน แต่ไม่ลงลึกถึง Technology Stack หรือ Physical Implementation (แม้ในไทยจะพบว่า SA ทำด้วยก็ตาม)
❌ ตาม Global Standard SA ทำ Logical Architecture ระดับ High-level เท่านั้น — Context Diagram, System Boundary, Integration Architecture ส่วน Physical Design เป็นงานของ Solution Architect และ Dev

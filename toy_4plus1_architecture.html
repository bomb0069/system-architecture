4+1 Architectural View Model — Philippe Kruchten (1995) แนวคิดที่ใช้ describe architecture จาก 5 มุมมอง ที่ตอบคนละกลุ่ม stakeholder — ไม่มี view เดียวที่บอกทุกอย่างได้ครบ architecture ที่ดีต้องสมเหตุสมผลในทุก view พร้อมกัน
5 Views ของ Architecture
🔄 กระบวนการทำงานกับ 4+1 Views (Iterative Process)
ไม่มี one-size-fits-all — โปรเจกต์ส่วนใหญ่เริ่มจาก Scenarios หรือ Logical ก่อน แล้วค่อย parallel กับ Process + Development และสุดท้าย Physical — แต่ระหว่างทาง ย้อนกลับมาปรับ view ก่อนหน้าได้เสมอเมื่อได้ข้อมูลเพิ่ม
① Scenarios — จุดเริ่มต้น
เก็บ Use Cases, User Stories, Quality Scenarios จาก stakeholders — เป็น input ของทุก view และเป็น test criteria สุดท้าย ถ้า architecture ตอบทุก scenario ได้ = architecture ใช้ได้
② Logical — กำหนด "อะไร"
ออกแบบ domain model, entities, relationships, responsibilities — ตอบคำถาม "ระบบประกอบด้วยอะไร ทำหน้าที่อะไร?" มักทำก่อนเพราะ Process และ Development ต้องอ้างอิง
③a Process — กำหนด "อย่างไร" (runtime)
Sequence diagrams, activity flows, concurrent processes — ตอบ "ใครคุยกับใคร ลำดับไหน?" มักทำ parallel กับ Development เพราะมีผลต่อกัน
③b Development — กำหนด "อย่างไร" (code)
Package structure, module boundaries, dependency rules — ตอบ "code จัดอยู่ที่ไหน ขึ้นกับอะไรได้?" กำหนด constraint ให้ developer ทีมโปรเจกต์ใหญ่ต้องทำชัดก่อน code
④ Physical — กำหนด "ที่ไหน + เท่าไหร่"
Infrastructure topology, deployment, scaling, HA/DR — ทำหลังสุดเพราะต้องรู้ว่า software structure เป็นอย่างไรก่อน จึงจะรู้ว่าต้องการ infra แบบไหน
⑤ Validate & Iterate
ตรวจว่าทุก view ตอบ scenarios ได้ครบไหม ถ้าไม่ — ย้อนกลับแก้ view ที่เกี่ยวข้อง ไม่ใช่แค่แก้ physical อย่างเดียว เป็น living document

💡 3 หลักการสำคัญของ 4+1 Model
1. Concern Separation
แต่ละ View ตอบคนละคำถาม — อย่าพยายามใส่ทุกอย่างใน diagram เดียว ถ้า diagram ตอบ 2 คำถามพร้อมกัน แสดงว่าควรแยกเป็น 2 view
2. Scenario Traceability
ทุก design decision ใน 4 views ต้องสืบย้อนกลับไปถึง Scenario ได้ ถ้า design ไม่มี Scenario รองรับ = อาจเป็น over-engineering
3. Consistency Across Views
Entity ใน Logical ต้องปรากฏใน Process ถูกต้อง, Module ใน Development ต้องปรากฏใน Physical ถูกต้อง — inconsistency คือ signal ว่า architecture ยังไม่สมบูรณ์

⭐ Loyalty Point — Full State Diagram (Extended)
Requirement ให้มาระบุแค่ Happy Path — แต่ในระบบจริงต้องออกแบบ state ให้ครอบคลุม ทุกสถานการณ์ที่อาจเกิดขึ้น ทั้งฝั่งลูกค้าและระบบ
● ACTIVE states ● TERMINAL states → transitions → error/edge cases
📋 State Transition Table — ออกแบบให้ครอบคลุม edge cases ทั้งหมด
STATE Trigger Actor แต้มใช้ได้? Next States Source

💡 Design Decision #1
Auto-Forfeit threshold: Requirement ไม่ได้ระบุว่าถ้าลูกค้าไม่กดยืนยันจะเกิดอะไร → ต้องนิยามเป็น configurable policy เช่น 30 วัน และต้องแจ้ง reminder ก่อน expire
💡 Design Decision #2
LOCKED state: ถ้าลูกค้าคืนสินค้าหลังกดรับ ต้องระงับแต้มก่อน (ไม่ revoke ทันที) เพราะ CS ต้องพิจารณาก่อน — ถ้า reject return แต้มต้องกลับมา ACTIVE ได้
💡 Design Decision #3
PARTIALLY_REDEEMED expiry: ถ้าแต้มหมดอายุขณะที่ใช้ไปแล้วบางส่วน → เฉพาะส่วนที่เหลือหมดอายุ ส่วนที่ใช้ไปแล้วไม่ได้รับผลกระทบ
💡 Design Decision #4
Earn calculation timing: แต้มคำนวณจาก order total หลังหักส่วนลดอื่นๆ แต่ก่อนหักแต้มที่ใช้ใน order นั้น — ป้องกัน circular dependency
